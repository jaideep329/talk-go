TalkGo CPU Throttling and Audio Jitter Notes

Why the Opus path existed, how Kubernetes CPU throttling works at 100m, and why it hurts a 20ms audio loop.

Brief Answer: Why Opus Encode/Decode Existed

Before the PCM-path change, Opus was used as an internal audio-frame format between TTSProcessor and PlaybackSinkProcessor. Cartesia gives us PCM. The old TTSProcessor.makeOpusData path chunked that PCM into 20ms frames and encoded each frame to Opus before pushing an AudioFrame downstream. Later, the old PlaybackSinkProcessor.decodeBotFrame path decoded the Opus back to PCM so it could mix bot speech with the required background-office audio and then send PCM to Daily through the Python bridge.

Implemented conclusion: the Opus step is not required by Daily egress. Daily's CustomAudioSource.write_frames accepts 16-bit linear PCM. For the current architecture, Opus was mostly an internal queue/packet format. Since playback immediately needs PCM again for mixing and Daily writing, the branch now keeps TTS audio as raw 20ms PCM frames inside Go.

What 100m CPU Means

Kubernetes measures CPU in CPU units. 1 CPU means one physical or virtual core's worth of CPU time, and 100m means 0.1 CPU, also described as 100 millicpu or 100 millicores. Kubernetes documents that 0.1 and 100m are equivalent CPU quantities. See Kubernetes: Resource units in Kubernetes.

A CPU request affects scheduling and relative CPU share under contention. A CPU limit is a hard cap. With a Linux cgroup CPU limit, the container gets a fixed CPU runtime budget in each CFS quota period. The Linux kernel CFS bandwidth documentation describes this as quota microseconds of CPU time inside each period; once quota is consumed, the cgroup is throttled until the next period. See Linux kernel: CFS Bandwidth Control.

The Usual 100ms Period

The common/default CFS period is 100ms. With that period:

Kubernetes CPU limit CPU quota per 100ms period Average CPU
100m 10ms CPU time 10% of one CPU
200m 20ms CPU time 20% of one CPU
500m 50ms CPU time 50% of one CPU
1000m 100ms CPU time 100% of one CPU
Important nuance: the quota is for the whole container cgroup, not for each process. Go, Python, Daily native threads, profiler threads, websocket readers/writers, and logging all spend from the same 10ms budget when the limit is 100m.

How Throttling Actually Happens

The kernel does not reserve a neat 2ms slice for every 20ms audio tick. It gives the cgroup up to its quota inside the current quota period. With 100m, the container can run for about 10ms of CPU time per 100ms period. If the container uses that budget early in the period, all normal CFS-scheduled threads in that cgroup stop running until the period refills.

Multi-threading can burn quota faster than wall-clock time. If two runnable threads each run for 5ms on two cores, the cgroup has spent 10ms CPU in only 5ms of wall time. That means a real-time audio loop may miss deadlines even when the average CPU looks close to, or slightly below, the limit.

What This Means For A 20ms Audio Loop

Our playback loop wants to do useful work every 20ms. Inside each 100ms CFS period there are about five such audio ticks. At 100m, those five ticks, plus Python's Daily write path, plus Go TTS work, plus all support goroutines/threads, share only about 10ms CPU total.

Time in one 100ms quota period Audio expectation CPU reality at 100m
0ms Audio tick 1 Go/Python can run if quota remains
20ms Audio tick 2 May run, or may already be throttled
40ms Audio tick 3 Likely vulnerable if TTS/bridge work used quota early
60ms Audio tick 4 If throttled, this tick is delayed until quota refill
80ms Audio tick 5 Same risk
100ms Next CFS period Quota refills and blocked threads can run again

This is why a 20ms audio loop can stutter even when the limit is described as "10% CPU." The audio loop is latency-sensitive, while CPU quota is an accounting mechanism over a larger period. The loop needs timely CPU access, not just acceptable average CPU over a second or a minute.

Why Jitter Appeared At The Beginning Of Bot Turns

The beginning of a bot turn is where multiple things become runnable together:

  1. LLM tokens start arriving.
  2. TTS starts producing PCM chunks.
  3. Go chunks those bytes into 20ms PCM frames.
  4. Playback converts PCM bytes to samples.
  5. Playback mixes bot PCM with required background-office audio.
  6. Go writes mixed PCM to the Python bridge every 20ms.
  7. Python calls Daily's native CustomAudioSource.write_frames.

That burst competes with the entire container's 10ms/100ms CPU budget. If the quota is consumed early, playback ticks are delayed. The user hears those delayed ticks as jitter.

Daily write_frames Boundary

The installed daily-python==0.27.0 package exposes CustomAudioSource.write_frames from a native extension named daily.abi3.so. There is no Python implementation to tune directly. The type stub says CustomAudioSource accepts 16-bit linear PCM, and the docstring says write_frames blocks unless a completion callback is supplied. 

def write_frames(
    self, frame: bytes, completion: Optional[Callable[[int], None]] = None
) -> int: ...

Docstring summary:
Writes audio frames to the audio source.
Blocks if a completion callback is not provided.
If a completion callback is provided, this function is non-blocking.

Daily's Python custom-track guide describes CustomAudioSource as the source you write audio data to for a custom audio track. See Daily Python SDK: Custom Tracks.

Implications For Optimization

Since background office audio is a hard requirement, removing continuous audio output is not on the table. The remaining useful optimization paths are:

  1. Remove the Opus round trip. This branch keeps bot audio as PCM in the Go pipeline, and lets playback mix PCM directly with background PCM. This removes Go's biggest historical hot path: opus.Encode, plus the corresponding decode.
  2. Experiment with fewer writes to Daily. If Daily accepts larger PCM chunks well, writing 40ms or 60ms at a time reduces calls through the Python/native boundary. This must be tested carefully because it may increase buffering latency.
  3. Consider non-blocking write_frames only with bounded backpressure. Passing a completion callback may prevent the Python stdin loop from blocking, but without a strict queue limit it can hide lag instead of removing it.
  4. Native Go Daily media is not a small optimization. Daily's supported headless server-media path here is the Python SDK backed by native core. A Go-native replacement would likely require unsupported bindings or a larger WebRTC/Daily implementation, and it would still have to do the native media work that write_frames is doing today.
