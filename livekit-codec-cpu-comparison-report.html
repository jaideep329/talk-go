Interpretation

  • PCMU is the lowest CPU result. It reduced steady Go CPU to 13.3m and removed Opus encode CPU entirely.
  • PCMU does not remove tick lag. Tick lag still appears because the remaining path is scheduling plus LiveKit/Pion/SRTP/UDP write pressure.
  • The lowest Opus CPU option is c1 / 20k / wideband. It is about 3m higher steady CPU than PCMU, but keeps Opus quality.
  • 100m has much lower throttling. The 100m runs show only 6-11ms steady throttling per 10s window, compared with roughly 130-170ms at 50m.
  • Background audio is not a major CPU lever. The no-background 100m run did not materially reduce steady Go CPU versus the comparable 100m Opus run.
  • Direct RTP writes may only be a small optimization. PCMU moved the bottleneck to WriteSample/RTP write, but most of that cost is the actual WebRTC write path rather than codec conversion.

Data Notes

  • All call metrics were taken from per-call Cloud Logging diagnostics and Pyroscope profiles.
  • Container-level aggregate CPU/memory averages were not used as primary evidence.
  • Steady throttling excludes the first 10 second startup bucket, which often includes room join and setup work.
  • Tick lag is a wall-clock measurement. CPU profiles identify what consumes CPU while running, but they do not directly show time spent waiting for CPU after quota throttling.
