Executive summary: PCMU with interim transcript RTVI events disabled is the lowest CPU option tested so far. It reduced steady Go CPU to 10.6m and cut LiveKit data publish volume from 4071 events to 204 events. It did not eliminate tick lag: this run had a 180ms tick-lag outlier even though playback work, write time, JSON work, and throttling were lower than the previous PCMU run. That points to scheduler/quota and turn-timing effects rather than codec work alone.

Interpretation

  • PCMU plus no interim events is the lowest CPU result. It reduced steady Go CPU to 10.6m and removed Opus encode CPU entirely.
  • Disabling interim transcript RTVI events helped signalling CPU. Data publishes dropped from 4071 to 204, JSON unmarshal dropped from 7.2% to 2.4%, JSON marshal dropped from 4.4% to 1.6%, and profile CPU dropped from 1.81s to 1.24s.
  • PCMU still does not remove tick lag. The latest run had lower playback work and throttling than the previous PCMU run, but a worse 180ms tick-lag outlier. That lag was not explained by PCMU conversion or normal write cost in the logs.
  • The lowest Opus CPU option is c1 / 20k / wideband. It is about 3m higher steady CPU than PCMU, but keeps Opus quality.
  • 100m has much lower throttling. The 100m runs show only 6-11ms steady throttling per 10s window, compared with roughly 130-170ms at 50m.
  • Background audio is not a major CPU lever. The no-background 100m run did not materially reduce steady Go CPU versus the comparable 100m Opus run.
  • Remaining CPU is mostly transport write path and runtime overhead. PCMU moved the bottleneck to WriteSample/RTP write; the no-interim change reduced app-message overhead, but the WebRTC write path remains the largest named CPU area.

Data Notes

  • All call metrics were taken from per-call Cloud Logging diagnostics and Pyroscope profiles.
  • Container-level aggregate CPU/memory averages were not used as primary evidence.
  • Steady throttling excludes the first 10 second startup bucket, which often includes room join and setup work.
  • Tick lag is a wall-clock measurement. CPU profiles identify what consumes CPU while running, but they do not directly show time spent waiting for CPU after quota throttling.
