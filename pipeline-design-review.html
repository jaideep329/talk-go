Section 0

Snapshot

One-liner: Your pipeline is a faithful but compact re-imagining of Pipecat's frame-graph, recast as one goroutine per processor with frames flowing through Go channels. The big trades you've made are in direction handling, system-frame priority, and shutdown semantics. None of them are wrong; all of them are worth knowing.

Pipecat processors (incl. Source/Sink, RTVI, aggregators)
10
Talk-Go processors (Source → AudioSrc → STT → Idle → Ctx → TalkTime → LLM → TTS → Playback → Sink)
2
Internal queues per Pipecat processor (priority input + process)
2
Channels per Talk-Go processor (Data + System), plus 1–5 internal channels each

Pipecat Mental model

Each processor is a self-contained async object with two internal queues and two asyncio tasks. process_frame(frame, direction) is the single entry point. Frames carry direction explicitly. Priority is enforced by a PriorityQueue.

Lifecycle is owned by a separate PipelineTask that pushes StartFrame, waits for EndFrame at the sink, then cleans up.

Talk-Go Mental model

Each processor is a single goroutine with a select over Data, System, and one-or-more internal IO channels. Direction is inferred from frame type. Priority is re-implemented as a nested select in every processor.

Lifecycle is owned by a Session that queues an EndFrame at the source and reacts when it reaches the sink. No StartFrame, no CancelFrame.

My take Where this review lands

You've kept the bones of Pipecat — frames, processors, source/sink edges, upstream/downstream, interrupts, graceful EndFrame — and dropped the parts you don't need yet (Observers, StartFrame/CancelFrame, nested pipelines, paused-state). That's a smart trim. The places I'd revisit are: routing (direction should be on the wire, not inferred), system-frame priority (extract the nested-select helper), and TTSProcessor state (consolidate the boolean salad into a small enum). Everything else is defensible.

Agree Single ContextAggregator

Pipecat's split (user before LLM, assistant after transport.output) exists because their context object is shared mutable state and they want each half positioned with the data flowing past it. Your single-aggregator design works because commit happens on upstream TTSDoneFrame, which reaches the same processor naturally. You get the same ordering guarantees with half the wiring. Keep it.

My take TalkTime position

Your placement lets it cancel the LLM via InterruptFrame downstream, which is what you want. Don't move it.

Section 2

Object model & possible states

Pipecat has three orchestration objects with explicit state machines (PipelineTask, Pipeline, FrameProcessor). You have one orchestration object (Session) plus stateless processors. The states you don't have aren't missing — they're either combined or implicit.

Pipecat objects

PipelineTask

Lifecycle owner. Holds push queue, observers, clock, task manager, idle & heartbeat monitors.

not-started → running → cancelling → finished

_pipeline_start_event, _pipeline_end_event, _pipeline_finished_event are asyncio.Events that gate transitions.

Pipeline

A FrameProcessor itself (composable, nestable). Auto-wraps your processors with PipelineSource + PipelineSink.

unset → setup() → cleanup()

No runtime state of its own; delegates to children.

FrameProcessor

Two internal queues, two asyncio tasks.

not-started → started → cancelling → cleaned-up

Sub-states: paused-frames, paused-system-frames. Plus _cancelling flag.

Talk-Go objects

Session

Conflates Pipecat's PipelineTask with external lifecycle (HTTP, UI WS, LiveKit room).

alive → ending → cleaned-up

Idempotency via endOnce & cleanupOnce (pipeline.go:78–85).

Pipeline

Just a wiring builder. Not a processor; not composable.

stateless

Run() spawns goroutines and returns immediately (pipeline.go:22–68).

FrameProcessor

Interface with a single Process(ProcessorChannels) method.

alive (goroutine running) → stopped (Process returns)

No formal state machine; each impl uses ad-hoc boolean flags (isStreaming, interrupted, awaitingTTSDone, pendingEnd, ending, botSpeaking, …).

My take Where Talk-Go saves complexity

You've correctly noticed that you don't need: StartFrame (because constructors do init work synchronously), StopFrame (because you only end gracefully), CancelFrame (same; with one 8-second escape hatch in TTS), paused/resumed (nothing pauses processors), and PipelineTask as a separate object (because per-session HTTP handlers wrap it). That's four state machines you've collapsed. The cost is paid in §7 (TTSProcessor) and §8 (lifecycle) — where the absence of those primitives forces local re-implementations.

Pushback Ad-hoc flags in processors

I'd flag TTSProcessor specifically. It currently carries five booleans (firstTextReceived, firstSentenceSent, firstAudioReceived, awaitingTTSDone, ttsFlushSent), a pointer (pendingEnd), an atomic, and a timer. That's a state machine wearing a costume. See §7 for the consolidation I'd suggest.

Section 3

Frame processor anatomy

Pipecat's processor is method-driven (one process_frame call per frame, framework runs two tasks behind it). Yours is loop-driven (one goroutine, one big select). The two designs do the same job, but they distribute responsibilities differently.

Same job, two shapes

Pipecat Method-driven

async def process_frame(self, frame, direction):
    await super().process_frame(frame, direction)
    if isinstance(frame, TranscriptionFrame):
        # do work
        await self.push_frame(out, direction)
    elif isinstance(frame, EndFrame):
        await self.cleanup()
        await self.push_frame(frame, direction)

The framework owns the loop, the queues, and the priority. You override one method, it's called once per frame.

Talk-Go Loop-driven

func (p *STTProcessor) Process(ch ProcessorChannels) {
    go p.readSTTWebsocket()
    go p.writeAudioWebsocket()
    for {
        select {
        case frame := <-ch.System:
            ch.Send(frame, Downstream)
        case frame := <-ch.Data:
            switch f := frame.(type) {
            case AudioFrame: ...
            case EndFrame:   ...
            }
        case frame := <-p.transcriptFrames:
            ch.Send(frame, Downstream)
        }
    }
}

You own the loop. Priority is a nested-select pattern you re-implement per processor. Background IO is funneled back through internal channels into the same select.

What each processor receives and sends

Pipecat Input & output

Input: the framework's __input_queue is a asyncio.PriorityQueue with two buckets — system frames (HIGH) and others (LOW). The __input_frame_task_handler pulls from it; system frames are processed inline, data frames are forwarded to __process_queue.

Output: push_frame(frame, direction) walks the _next/_prev pointer set during Pipeline._link_processors and calls the neighbor's queue_frame().

Input: two buffered channels per processor — Data (size 100) and System (size 100). Priority is enforced manually inside Process with a nested select (system first; default drops into a wider select).

The shape of "Process" across your processors

Every Process loop has the same skeleton — they vary only in which internal channels they add and which frame types they intercept:

for {
    select {
    case frame := <-ch.System:           // priority
        // handle interrupt
    default:
        select {
        case frame := <-ch.System:       // duplicate, for fairness
        case frame := <-ch.Data:         // normal work
        case ev := <-p.internalCh:       // async IO fed back in
        case <-ticker.C:                  // playback only
        case <-pendingEndTimer.C:         // tts only
        }
    }
}
Pipecat Why two tasks instead of one

Pipecat splits ingestion and processing into two tasks so that: (a) System frames can be handled even while a data frame is being processed — the input task pulls from the priority queue and dispatches system frames inline without waiting for the slow-running process task; and (b) On interruption, the process task is cancelled and recreated, which atomically purges queued data frames while keeping UninterruptibleFrames. You can't do either of those with a single goroutine.

Talk-Go Why one goroutine works for you

You don't have long-running synchronous work inside a processor — every blocking IO is off-goroutine (websocket reader/writer, RTP track reader). So your select is almost always near-idle and a priority check via nested select gives the same effective behavior as a real priority queue. Interrupts get processed within microseconds of arriving.

My take The duplication smell

The nested-select pattern is correct but copy-pasted across seven processors. A single helper — call it nextFrame(ch) (Frame, Direction, bool) — would let every processor read for { f, dir, ok := nextFrame(ch); if !ok { return }; switch f.(type) {…} }. You'd delete ~80 lines without changing behavior. See §5.

Section 4

Frame routing & direction

Pipecat carries direction with each frame through the queue tuple. Talk-Go discards direction at the channel boundary; processors reconstruct it from the frame's type. That implicit convention works today, but it's the source of ~30 lines of pass-through boilerplate and silently rules out broadcast frames.

Where direction lives

Pipecat Direction is data

# input_queue holds tuples
await super().put((PRIORITY, counter, (frame, direction, callback)))

# neighbor pointers
self._prev: Optional[FrameProcessor] = None
self._next: Optional[FrameProcessor] = None

# push_frame
await self._next.queue_frame(frame, direction)

Each processor knows its neighbors. Direction is always explicit at the call site.

Talk-Go Direction is type-inferred

// channel holds bare frames; direction is lost
type ProcessorChannels struct {
    Data   <-chan Frame
    System <-chan Frame
    Send   func(frame Frame, dir Direction)
}

// central router knows positions, not processors
sendFn := func(i int) func(Frame, Direction) {
    return func(frame Frame, dir Direction) {
        switch dir {
        case Downstream: /* route to i+1 */
        case Upstream:   /* route to i-1 */
        }
    }
}

Processors don't know their neighbors. They read frames; whether a frame came from above or below is something they decide from its Go type.

My take Put direction on the wire

Wrap channel elements in an Envelope{Frame, Direction}. Then the per-processor switch ends with a one-liner default: ch.Send(env.Frame, env.Direction). The twelve pass-through cases vanish. You also unlock broadcast (one frame, sent both ways from the same call site) and upstream system frames (which your current router silently routes through the data channel instead).

Pipecat Reference: broadcast_interruption

Pipecat exposes broadcast_frame and broadcast_interruption on FrameProcessor (frame_processor.py:763–768, 790–806). Both push the same frame to _next and _prev. The implementation even sets broadcast_sibling_id on each copy so observers can match the pair. That capability simply isn't expressible in your current router.

Talk-Go Why the implicit table works today

None of the frame types you've defined currently need to travel in both directions. BotStartedSpeakingFrame only goes up (PlaybackSink emits it; you don't ever push it down). InterruptFrame only goes down. So the convention holds. The first time you want something to broadcast, you'll have to either add a sentinel or migrate to Envelope.

Section 5

System-frame priority

Same goal, very different implementations. Pipecat uses a PriorityQueue; you use a nested select in every processor. Both work. The latter is the most duplicated pattern in your codebase.

Pipecat PriorityQueue

class FrameProcessorQueue(asyncio.PriorityQueue):
    HIGH_PRIORITY = 1
    LOW_PRIORITY  = 2

    async def put(self, item):
        frame, _, _ = item
        if isinstance(frame, SystemFrame):
            await super().put((HIGH, ctr, item))
        else:
            await super().put((LOW, ctr, item))

The queue handles ordering. Each processor sees frames in priority order. Lifecycle handlers don't need to know anything about priority.

frame_processor.py:85–133

Talk-Go Nested select

for {
    select {                            // priority pass

The helper extraction

Sketch — once written, every processor's loop becomes one level flatter:

// frame_processor.go (helper)
func nextFrame(ch ProcessorChannels) (Frame, Direction, bool) {
    select {
    case f, ok := <-ch.System:
        if !ok { return nil, 0, false }
        return f.Frame, f.Direction, true
    default:
        select {
        case f, ok := <-ch.System:
            if !ok { return nil, 0, false }
            return f.Frame, f.Direction, true
        case f, ok := <-ch.Data:
            if !ok { return nil, 0, false }
            return f.Frame, f.Direction, true
        }
    }
}

// in each processor
for {
    f, dir, ok := nextFrame(ch)
    if !ok { return }
    switch v := f.(type) {
    // only the cases this processor cares about
    default:
        ch.Send(f, dir)
    }
}

The wrinkle: processors with extra internal channels (TTS's ttsEvents, Playback's ticker, etc.) still need a select. Two ways to handle that:

  1. Keep nextFrame for "vanilla" processors; non-vanilla ones write the select manually but still use a helper for the priority pass.
  2. Generalize: nextFrame(ch, extras...) takes additional cases via a slice of reflect.Select cases. (I'd not go this far — too much machinery for three callers.)

You chose: extract. Recommended order in §12 — do this first (mechanical, no behavior change).
Section 6

Interruption flow

Pipecat broadcasts InterruptionFrame both directions and automatically purges queued data frames (except UninterruptibleFrames). You send a single downstream InterruptFrame and each processor manually clears its own buffers. There's one subtle invariant you're relying on that Pipecat enforces structurally.

Pipecat broadcast_interruption

  1. Some processor calls broadcast_interruption() on itself.
  2. An InterruptionFrame (a SystemFrame) is queued both upstream and downstream.
  3. Every processor's process_frame sees the system frame priority-first. The base class calls _start_interruption() which cancels and recreates the process task.
  4. The recreate step purges __process_queue — any pending data frames are dropped except UninterruptibleFrames (EndFrame qualifies).
  5. Metrics auto-stop. The interrupt continues propagating.
Source pointers
frame_processor.py:763–768 (broadcast_interruption) · frame_processor.py:892–907 (_start_interruption) · frame_processor.py:995–1012 (__reset_process_queue — keeps UninterruptibleFrame) · frames.py:1877 (EndFrame is UninterruptibleFrame)

Talk-Go Downstream InterruptFrame

  1. ContextAggregator.Process detects barge-in (3 interim words while botSpeaking) and calls ch.Send(InterruptFrame{}, Downstream) (context_aggregator.go:160–168).
  2. Each downstream processor receives InterruptFrame on its System channel, sees it via the priority pass, manually clears its own state (e.g. p.CancelTTSContext(); clearTTSState(); drainTTSEvents(t.ttsEvents)), and forwards downstream.
  3. No upstream propagation. STT/AudioSource keep running undisturbed.
  4. Each buffer-owning processor (TTS, Playback) has its own clearing code; there's no automatic purge.

The invariant you're depending on

Pushback EndFrame ↔ InterruptFrame race

On InterruptFrame, PlaybackSinkProcessor sets p.playbackQueue = nil (playback_sink_processor.go:192–202). If an EndFrame were already in playbackQueue at that moment, it would be lost.

In your current topology this can't actually happen: the only producers of EndFrame are Session.End() (via PipelineSource) and TalkTimeMonitor. Neither runs concurrently with a barge-in interrupt that could reach PlaybackSink. But the invariant is informal. Pipecat encodes it structurally: EndFrame is UninterruptibleFrame, and __reset_process_queue keeps those frames during a purge.

Cheap fix in your code: when clearing playbackQueue, walk it and preserve any EndFrame:

case InterruptFrame:
    p.interrupted = true
    keep := p.playbackQueue[:0]
    for _, f := range p.playbackQueue {
        if _, isEnd := f.(EndFrame); isEnd {
            keep = append(keep, f)
        }
    }
    p.playbackQueue = keep

No new abstractions, no UninterruptibleFrame mixin needed. Just one targeted safety net.

Why one-way interrupt is fine for you

Agree Downstream-only is correct here

Pipecat broadcasts because their upstream processors (input transport, VAD) need to react — they may have queued speech frames that should be flushed. In your topology, AudioSource and STT don't queue anything that needs purging on barge-in: STT keeps streaming user speech (that's literally what caused the barge-in), and AudioSource is a thin opus→PCM pipe. So upstream propagation would be a no-op.

You correctly handle the one case where upstream-ish reaction is needed — UserIdleProcessor needs to know the bot stopped speaking — via BotStoppedSpeakingFrame traveling upstream from PlaybackSink, not via the interrupt itself.

Section 7

TTS & EndFrame drain

This is the most complex piece in your codebase, and most of the complexity is intrinsic. The drain has to live in TTS because TTS is the only place that knows when Cartesia's async stream has finished. I'll show why your simpler-looking alternative (move drain to PlaybackSink) doesn't work, then sketch the targeted simplification that does.

Why this exists at all

On EndFrame, TTS may be holding:

  • Unsent text (aggregated tokens waiting for a sentence boundary)
  • In-flight synthesis: text already sent to Cartesia, audio still streaming back over the websocket
  • Buffered PCM not yet packed into a full 20ms opus frame
  • Pending word timestamps not yet emitted (waiting for audioTimePushed to catch up)

All four must drain before EndFrame can propagate downstream — otherwise PlaybackSink would receive EndFrame before the final audio, the silence tail would play before the last syllable, and the call would cut off mid-word.

Pipecat's drain — split between TTS service and output transport

Pipecat TTS service

The TTS service (CartesiaTTSService etc.) is responsible for sending unsent text and emitting TTSStartedFrame/TTSAudioRawFrame/TTSStoppedFrame. It does not buffer encoded audio — chunks flow through immediately.

Pipecat output transport

BaseOutputTransport.MediaSender owns an asyncio.Queue of audio frames. On EndFrame, it queues the EndFrame at the back of the audio queue and the audio task drains in order; when it pops the EndFrame it writes audio_out_end_silence_secs of silence and exits (base_output.py:794–821).

Crucially: EndFrame is UninterruptibleFrame so it survives interruption purges and is never reordered.

Why you can't just "move drain to PlaybackSink"

This was my first instinct (and my first message suggested it). On closer inspection it breaks ordering. The boundary between "async Cartesia chunks arriving via websocket" and "ordered downstream frames" is TTS itself. If TTS forwarded EndFrame the moment it received it, the timeline would look like this:

t = 0ms
TTS reads EndFrame from Data channel. Forwards it downstream immediately.
t = 0–500ms
Cartesia is still streaming chunks for the previous turn. They arrive on the websocket → ttsEvents → main goroutine → forwarded as AudioFrame downstream.
t = 500ms
Cartesia sends done. TTS forwards TTSDoneFrame downstream.
— PlaybackSink sees, in order: —
EndFrame first, then many AudioFrames, then TTSDoneFrame. Wrong order. PlaybackSink would play the silence tail before the audio.

The only fix is for TTS to not forward EndFrame until it has emitted the final TTSDoneFrame. Which is exactly what it does today. So the drain has to stay in TTS.

Pushback (on my earlier suggestion)

In my first response I floated "move drain to PlaybackSink" as option 4d. I'm retracting it. The structural reason it doesn't work is above. The drain has to live in TTS.

Recommended consolidation

type ttsSynth int
const (
    ttsSynthIdle      ttsSynth = iota  // no synthesis in flight
    ttsSynthStreaming                  // text sent, audio coming back
    ttsSynthClosing                    // ResetTTSContext sent; waiting for last 'done'

type TTSProcessor struct {
    // metrics one-shots stay as-is — they're unrelated
    firstTextReceived, firstSentenceSent, firstAudioReceived bool

    // the lifecycle, named
    synth      ttsSynth
    pendingEnd *EndFrame
    endTimer   *time.Timer
    ...
}

Each event becomes a phase transition:

  • LLMResponseStart / TTSSpeakFrame: ttsSynthIdle → ttsSynthIdle (no change; start aggregating).
  • TextFrame ending in punctuation, sent: ttsSynthIdle → ttsSynthStreaming.
  • LLMResponseEnd: flush + ResetTTSContext: ttsSynthStreaming → ttsSynthClosing.
  • Cartesia done event: ttsSynthClosing → ttsSynthIdle; forward TTSDoneFrame.
  • EndFrame:
    • If ttsSynthIdle: forward immediately, return.
    • Else: stash in pendingEnd, ensure we're in ttsSynthClosing, start endTimer.
  • InterruptFrame: cancel context; ttsSynth* → ttsSynthIdle; drainTTSEvents.

The if pendingEnd != nil { /* drop frames */ } branch can be expressed as if pendingEnd != nil && !isUpstream(env.Direction) { drop } if you also adopt explicit direction (§4). That alone removes ~15 lines from the Process loop.

at PlaybackSink
The queue contains: AudioFrame×N, WordTimestampFrame×M, TTSDoneFrame, EndFrame. Ticker drains them in order at 20ms pace; on hitting EndFrame, writes playbackEndTail of silence, forwards EndFrame downstream.

My take Net win, but bounded

The consolidation removes ~30 lines from tts_processor.go and makes the state machine readable in a glance. The drain logic itself doesn't get simpler — it can't, because the underlying ordering constraint is fundamental. What gets simpler is reading the code.

Pipecat parallel

Pipecat's equivalent of "ResetTTSContext to elicit a final done" is the TTS service's flush() coroutine called on EndFrame. Their state machine isn't simpler than yours — they just spread it across two classes (service and transport sender) instead of one. The 8s escape hatch is your own; Pipecat relies on CancelFrame to forcibly stop the pipeline after CANCEL_TIMEOUT_SECS = 20.

Section 8

Lifecycle, shutdown, & the missing StartFrame

Pipecat has four lifecycle frames (StartFrame, StopFrame, EndFrame, CancelFrame). You have one (EndFrame) plus ctx-cancellation. That collapse is mostly fine, but it shows up in two places: /connect HTTP latency and stuck-shutdown recovery.

Lifecycle frames, compared

The shutdown-resource invariant

Pushback Add a defer for safety

Your invariant — "resources owned by a processor are closed inside its Process goroutine, only when EndFrame passes through" — is informal. AGENTS.md states it; the code doesn't enforce it. Two places it could fail:

  • If sessionCtx.Ctx is cancelled before EndFrame reaches a processor, the processor's main Process loop doesn't exit — only inner reader goroutines do. The websocket may hang in ReadMessage until the connection times out.
  • If Process() panics for any reason, the websocket is leaked.

Two minimal changes:

func (p *STTProcessor) Process(ch ProcessorChannels) {
    defer p.stop()                                 // idempotent via sync.Once
    go p.readSTTWebsocket()
    go p.writeAudioWebsocket()
    for {
        select {
        case <-p.sessionCtx.Ctx.Done():            // last-resort exit
            return
        case ...
        }
    }
}
