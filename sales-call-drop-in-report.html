Two hard blockers remain for a true production cutover. First, Python starts the sales call by queuing the seeded LLM context on Daily on_client_connected; Go currently stores the seed ({user: "hello?"}) but never queues it, so fresh calls likely wait for the user to speak first. Second, the LLM swap is still open: Go uses OpenAI gpt-4.1 with no temperature/switching/entity tracing, while production runs Grok-4.1-fast on Azure with temperature=0 and failover. The rest are operational/data parity items: a global 120 s no-frames watchdog, per-conversation Cartesia key, RTVI bot-ready/client-ready handling, and a few exact-shape differences in debug logs and edge-case prompts.

1. Scope & method

I read the current talk-go tree (sources and tests, plus AGENTS.md), then walked the Python sales-call from bots/sales_call/sales_call.py outward into:

  • bots/onboarding_call/onboarding_call_init_manager.py — the cache + DB fetch for conversation data & user profile.
  • bots/followup_call/fetch_conversation.py — the shared loader that builds the resume-message + unprocessed-chat-context.
  • bots/onboarding_call/conversation_persistence_processor.py — per-turn Redis writes + S3 upload + chunk-sync enqueue.
  • bots/onboarding_call/custom_rtvi_observer.py — RTVI debug-log capture & first-speech timestamping.
  • bots/onboarding_call/custom_daily_transport.py — Daily join retry + first-audio timestamp.
  • bots/onboarding_call/custom_soniox_stt.py, phonetic_text_filter.py, metrics_tracker.py, talktime_monitor.py.
  • bots/bot_worker_manager.py, bots/signal_handler.py, bots/gke_pod_manager.py (worker registration, pod safe-to-evict, cleanup), bots/views.py/urls.py (worker HTTP API).
  • bots/services/voice_bot_api_service.py — /bot/update_conversation, /bot/run_post_call_operations contract.
  • services/conversation_chunk_manager.py + services/conversation_chunks_cache.py — Redis key shape, sync job.
  • common/sqs_queue_mapping_manager.py — queue names.

For each component I asked three questions: does talk-go reach byte-for-byte parity, does it reach semantic parity (good enough), or is it missing? Behavioural differences are weighted by user impact (does it change what the sales user hears) and by ops impact (does it leak a pod, lose a metric, etc).

3. Inside the bot — sales_call.py vs SalesCallBot

3.1 Entry & conversation load

Python: run_bot(room_url, token, conversation_id)

  • Calls fetch_conversation('sales_call', conversation_id) which:
    • Reads conversation_data:{cid} from Redis (DB fallback).
    • Loads UserProfile from cache or DB.
    • Pulls unprocessed_chat_context (cache or DB) and appends to short_term_memory.
    • Constructs resume_message if resumed_from_chunk_id is set.
    • Returns 21 fields including patient_info, profile_id, direction, gender, campaign_pricing_experiment_flag, phone, compiled_call_flow_s3_key, etc.
  • Calls get_sales_call_user_profile_data for remaining_sales_call_talktime_seconds; defaults to LIFETIME_TALKTIME_LIMIT_SECONDS when null.

Go: SalesCallBot.plan / BuildTask

  • collectCallStartup reads conversation_data:{cid} from Redis only (no DB fallback) and validates bot_type.
  • Resolves user_id from conversation.user_id or user_profile.user_id.
  • Now reads the full sales-call surface from the same Redis payload: patient_info, resumed_from_chunk_id, resume_gracefully, resumed_chunk, unprocessed_chat_context, campaign_pricing_experiment_flag, remaining_sales_call_talktime_seconds, short_term_memory.
  • loadSalesPrompt picks the Langfuse prompt name by experiment flag and fetches it via the new disha.DocumentStore (Redis only).
  • buildResumeSystemMessage appends the 5-min/>5-min Hindi resume preamble when resumed_from_chunk_id + resumed_chunk are present.
  • Wires opts.TTSFilter = deps.PhoneticDict so every Cartesia send goes through the phonetic dictionary.
What still differs in this section Talk-go does not query the Disha DB on cache miss (by design — workers have no DB connection). If conversation_data:{cid} isn’t in Redis, SalesCallBot.plan fails the /connect with an error, just like a Disha worker missing the prefetch would. Disha’s main app is responsible for warming that key before invoking create_worker_room; this already happens today.

3.2 STT

Python

CustomSonioxSTTService(api_key=…, vad_force_turn_endpoint=False, params=SonioxInputParams(language_hints=[Language.HI])) — subclass adds tenacity retry on initial connect and exits with sys.exit(1) when exhausted; emits Sentry.

Go

STTProcessor dials directly to wss://stt-rt.soniox.com with model: "stt-rt-v4", language_hints: ["hi"], enable_endpoint_detection: true, max_endpoint_delay_ms: 300. Lazy connect, automatic reconnect on read error, and Sentry capture when the capped initial connect attempts are exhausted.

Functionally equivalent for the happy path. The initial connect retry cap is now also equivalent: Go tries three times and then emits a fatal ErrorFrame instead of holding the worker slot forever.

3.3 LLM

Python

CustomOpenAILLMService(model="grok-4-1-fast-non-reasoning", api_key=GROK_4_1_FNR_EASTUS_API_KEY, base_url="https://eastusresourcegroup.openai.azure.com/openai/v1", deployment="cureai-grok4fnr-eastus", switch_llm_config=True, model_group='grok-4.1-fast-sales'). Carries user_id+entity_id for LLM logging, with automatic switching via LLMSwitchingService on errors, temperature=0, prompt metadata persisted to chunks.

Go

LLMProcessor uses hard-coded llmModel = "gpt-4.1" against https://api.openai.com/v1/chat/completions with OPENAI_API_KEY. No temperature control, no Langfuse linkage, no fallback model, no per-call entity tracing.
Hard blocker for drop-in Switching the production sales-call from Grok-4.1-fast (an Azure deployment that’s actively price- and quality-tuned for sales) to OpenAI gpt-4.1 will change every conversion-rate, latency, and cost number. LLMSwitchingService also handles model failover; without it any Grok outage today would have nowhere to switch to. This must be parameterised before cutover.

3.5 Conversation context & initial messages

Python

  1. If previous_messages: rebuild as {role, content} filtering is_debug_log and additional_data.
  2. Prepend system prompt with patient_info+current_datetime substitution.
  3. If resume_message: append a fresh system message.
  4. Otherwise (fresh call): seed with system prompt + {user: "hello?"}.
Wraps the result in LLMContext with system_message_prompt_key, prompt_metadata, system_message_params — these are persisted with every chunk.

Go

initialSalesMessages(prompt, startup.Data.Chunks, resumeMsg) mirrors all four steps:
  1. Replays non-debug chunks via messageFromChunkTuple.
  2. System prompt comes from DocumentStore.GetDocument(name, 0, vars) — vars contains patient_info, current_datetime, short_term_memory.
  3. Appends the resume system message when buildResumeSystemMessage returns non-empty.
  4. Fresh call seeded with {user: "hello?"} when no prior chunks were replayed.
Done: system_message_prompt_key persistence on chunks. The document version is available via PromptKey(name, version), passed into ContextAggregatorConfig, then supplied to the committed-turn callbacks when chunks are written.
New hard gap found in this audit The context is seeded, but it is not kicked off. Python queues main_context_aggregator.user()._get_context_frame() from on_client_connected, which makes the bot speak first from {user: "hello?"}. Go only emits LLMMessagesFrame after a user transcript reaches ContextAggregator. Add a one-shot initial-context queue when the Daily user/client connects, preferably owned by the Disha sales-call assembly rather than the core framework.

3.6 Talk-time monitor

Python

TalktimeMonitor arms its timer only after the first UserStartedSpeakingFrame. On timeout: pushes TTSSpeakFrame("Your allocated talk time has been exhausted. Thank you for your time."), calls on_talktime_exhausted (PATCH /update_conversation with end_reason, send RTVI talktime_exhausted, perform_cleanup).

Go

TalkTimeMonitoringProcessor now waits on a startUserCh channel that closes on the first LLMMessagesFrame (which ContextAggregator only emits after a committed user turn). Until then the goroutine sits idle. On timeout it emits InterruptFrame + TTSSpeakFrame("Your talk time is exhausted now. Ending the call.") downstream and calls taskCtx.EndTask(EndReasonTalkTimeExhausted). Exact parity would use Python’s exhaustion sentence and arm on first speech rather than committed turn.

3.7 User idle

Python

UserIdleProcessor from Pipecat with timeout=5.0, callback handle_idle that pushes TTSSpeakFrame("hello?", append_to_context=True) up to 6 times and then calls task.cancel() with user_idle_timeout_triggered=True → end_reason user_idle on the post-call.

Go

UserIdleProcessor: 7 s timeout, max 7 prompts. The cap fire now speaks the prompt one final time and then calls taskCtx.EndTask(EndReasonUserIdle) so the EndFrame routes through the pipeline source — end_reason ends up as "user_idle" on the post-call write, matching Python. Exact timing/text parity would be 5 s and lowercase "hello?".

3.8 RTVI & client messages

Python

CustomRTVIProcessor + CustomRtviObserver emit rtvi-ai entries for bot-transcription, user-transcription, bot-started/stopped-speaking, server-message (metrics, llm_call_result, interruption, talktime_exhausted, participant-left). Also handles inbound {type:"ping"} and responds with "pong" via send_server_response.

Go

UIEventSender emits all the outbound types above through Daily app messages and snapshots them into the S3 debug log. Inbound side handles {type:"end_call"} plus RTVI client-message ping with a server-response pong. Missing: client-ready → bot-ready, and Go currently includes interim user-transcription entries in the same buffered S3 stream.

RTVI pong-based keepalive is now covered for clients that send client-message pings over Daily app messages. Bot-ready remains open for strict RTVI clients.

4. Pipeline parity (frames, processors, ordering)

Both run the same logical chain:

transport.input() → user_idle → rtvi → stt → user_aggregator → llm → talktime → tts → transport.output() → assistant_aggregator

The Go ordering folds the user and assistant aggregators into one ContextAggregator that lives between STT and TalkTime — functionally equivalent because the Go aggregator commits on TTSDoneFrame arriving upstream from PlaybackSink. Differences worth flagging:

  • Min-words barge-in — both use min_words=3. ✓
  • SpeechTimeoutUserTurnStopStrategy (Python user_speech_timeout=0.05) — talk-go relies on Soniox's <end> token directly; functionally similar but the 50 ms guard does not exist.
  • Back-channel discard — talk-go implements both branches (in-progress and at-bot-turn-end), matching Pipecat reset_aggregation. ✓
  • InterruptFrame purges procCh keeping !IsInterruptible() — mirror of Pipecat UninterruptibleFrame mixin. ✓
  • WordTimestampFrame interleaving — both interleave words with audio frames at PlaybackSink; talk-go uses Cartesia add_timestamps. ✓
  • EndFrame-driven cleanup — both route shutdown through the source so STT/AudioSource/UserIdle/ContextAggregator see it in pipeline order. ✓
  • Custom Daily transport join retry (3×) — Python implements CustomDailyTransportClient.join with 3 retries. talk-go's JoinDailyRoom now matches that retry behavior around the bridge join.
  • Background office sound mixer — ✓ now confirmed wired. PlaybackSinkProcessor calls loadBackgroundPCM("background-office-sound.mp3") and mixPCM blends it under the bot speech (bgVolume=0.5, botVolume=0.3) — the Go equivalent of Python's CustomSoundMixer(... volume=0.5).
  • Pipeline assembly location — ✓ talk-go now builds the processor list inside the bot (SalesCallBot.BuildTask) and hands it to a bare PipelineTask, mirroring how sales_call.py constructs Pipeline([...]) in run_bot. voicepipelinecore exposes building blocks (NewPipelineTask, processor constructors, SetPipeline) rather than an all-in-one factory.

6.4 Why safe-to-evict matters

GKE Autopilot uses the cluster-autoscaler.kubernetes.io/safe-to-evict annotation to decide whether a pod can be evicted during scale-down. Python flips it to false at two moments: when the pod is reserved, and when the pod becomes active for a call. talk-go now matches both — pinPodAgainstEviction() runs from mark_machine_reserved (covering the several-second window between Disha's reserve_worker_pod and create_worker_room, before the user joins) and again at call start (covering the direct /connect path), then unpinPodAfterCall() restores true on cleanup. Both calls are idempotent and no-op outside Kubernetes.

One residual edge A pod that is reserved but never receives a call stays pinned until it exits, since the unpin is tied to call cleanup. This matches Python and is safe (the annotation only constrains a still-running pod); the SIGTERM / state-manager path reaps abandoned reservations.

7. Signal handling & graceful shutdown

Both implementations follow the same protocol:

  1. First SIGTERM — ignore duplicates, write Redis key pod_sigterm:{pod} with 2 h TTL, enqueue bots.signal_handler.on_graceful_shutdown_initiated on fifo-p0-fast-l1.
  2. Idle pod → exit immediately so Kubernetes rollouts don’t stall.
  3. Active/reserved pod → stay alive; Disha’s state-manager job decides whether to cleanup.
  4. SIGINT → immediate exit for local dev parity.

talk-go’s registerCleanupHandlers matches this. Differences:

  • Python has an atexit handler that emits a Sentry "Abrupt machine kill detected" message when shutdown wasn’t graceful. talk-go now has reportAbruptShutdownOnExit, keyed by FLY_MACHINE_ID/HOSTNAME when present.
  • Python uses os.write(2, ...) for signal-safe logging (avoids reentrancy into buffered I/O); Go’s log package is already safe enough that this is not needed.

Where talk-go is genuinely better Independent of parity, the Go pipeline has cleaner cancellation semantics than the Pipecat 0.0.108 build (explicit per-processor procCtx, EndFrame survives interrupt purges via IsInterruptible(), single-shared WaitGroup with bounded wait + goroutine dump on timeout). Replacing the Python sales-call is not a downgrade once M1–M9 are in. The process model — one binary per pod, no asyncio scheduler under load — should give you a measurable v2v p99 improvement on busy machines.
