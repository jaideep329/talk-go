talk-go Sales Call — Drop-In Readiness Report

A side-by-side review of the current talk-go implementation against Disha-Backend's bots/sales_call/sales_call.py and every surrounding operation: post-call persistence, worker machine lifecycle, signal handling, and the conversation context surface. Use this to decide whether talk-go can replace the Python sales-call worker today, and to scope the remaining work if not.

Generated: Wed 27 May 2026 talk-go @ /Users/jaideepsingh/Projects/talk-go disha-backend @ /Users/jaideepsingh/Projects/disha-backend Pipecat reference: pipecat-ai==0.0.108

Verdict — Not yet a 1:1 drop-in for production sales calls

Pipeline shape: ready Persistence: partial Prompt / LLM: gap User-experience polish: gap

The Pipecat-style frame pipeline, Disha API contract, Redis chunk shape, worker lifecycle, signal handling and SQS-job wiring are all faithfully mirrored. What's missing is the business surface of the Python bot: the dynamic Langfuse prompts gated on campaign_pricing_experiment_flag, the Grok-4.1-fast Azure LLM (talk-go still uses gpt-4.1), the Hindi PhoneticTextFilter for Cartesia, the resume_message/unprocessed_chat_context path, and a handful of bug-shaped behavioural differences (idle never ends the call, talktime starts on construction not first user speech, no Sentry, no GKE safe-to-evict annotation, etc). Concretely — pointing Disha’s create_room at talk-go today will connect, talk and persist, but with the wrong prompt, the wrong LLM, an English voice for a Hindi user, no abuse recovery, and a few edge cases that may keep a pod alive longer than it should.

1. Scope & method

I read the current talk-go tree (sources and tests, plus AGENTS.md), then walked the Python sales-call from bots/sales_call/sales_call.py outward into:

  • bots/onboarding_call/onboarding_call_init_manager.py — the cache + DB fetch for conversation data & user profile.
  • bots/followup_call/fetch_conversation.py — the shared loader that builds the resume-message + unprocessed-chat-context.
  • bots/onboarding_call/conversation_persistence_processor.py — per-turn Redis writes + S3 upload + chunk-sync enqueue.
  • bots/onboarding_call/custom_rtvi_observer.py — RTVI debug-log capture & first-speech timestamping.
  • bots/onboarding_call/custom_daily_transport.py — Daily join retry + first-audio timestamp.
  • bots/onboarding_call/custom_soniox_stt.py, phonetic_text_filter.py, metrics_tracker.py, talktime_monitor.py.
  • bots/bot_worker_manager.py, bots/signal_handler.py, bots/gke_pod_manager.py (worker registration, pod safe-to-evict, cleanup), bots/views.py/urls.py (worker HTTP API).
  • bots/services/voice_bot_api_service.py — /bot/update_conversation, /bot/run_post_call_operations contract.
  • services/conversation_chunk_manager.py + services/conversation_chunks_cache.py — Redis key shape, sync job.
  • common/sqs_queue_mapping_manager.py — queue names.

For each component I asked three questions: does talk-go reach byte-for-byte parity, does it reach semantic parity (good enough), or is it missing? Behavioural differences are weighted by user impact (does it change what the sales user hears) and by ops impact (does it leak a pod, lose a metric, etc).

3. Inside the bot — sales_call.py vs SalesCallBot

3.1 Entry & conversation load

Python: run_bot(room_url, token, conversation_id)

  • Calls fetch_conversation('sales_call', conversation_id) which:
    • Reads conversation_data:{cid} from Redis (DB fallback).
    • Loads UserProfile from cache or DB.
    • Pulls unprocessed_chat_context (cache or DB) and appends to short_term_memory.
    • Constructs resume_message if resumed_from_chunk_id is set.
    • Returns 21 fields including patient_info, profile_id, direction, gender, campaign_pricing_experiment_flag, phone, compiled_call_flow_s3_key, etc.
  • Calls get_sales_call_user_profile_data for remaining_sales_call_talktime_seconds; defaults to LIFETIME_TALKTIME_LIMIT_SECONDS when null.

Go: SalesCallBot.BuildOptions

  • collectCallStartup reads conversation_data:{cid} from Redis only (no DB fallback) and validates bot_type.
  • Resolves user_id from conversation.user_id or user_profile.user_id.
  • Reads only remaining_sales_call_talktime_seconds; everything else in the rich Python payload is ignored.
  • No unprocessed_chat_context, no resume_message, no patient_info, no direction, no campaign_pricing_experiment_flag.
Why this matters A sales-call user who retries within 5 minutes will land in talk-go without the “hanji to aap keh rhe the” resume preamble. New users will not get coupon-experiment prompts. Health context from unprocessed_chat_context (e.g. a chat message they just sent) never reaches the LLM.

3.2 STT

Python

CustomSonioxSTTService(api_key=…, vad_force_turn_endpoint=False, params=SonioxInputParams(language_hints=[Language.HI])) — subclass adds tenacity retry on initial connect and exits with sys.exit(1) when exhausted; emits Sentry.

Go

STTProcessor dials directly to wss://stt-rt.soniox.com with model: "stt-rt-v4", language_hints: ["hi"], enable_endpoint_detection: true, max_endpoint_delay_ms: 300. Lazy connect, automatic reconnect on read error. No Sentry; logs warnings only.

Functionally equivalent for the happy path. Difference: connect failures retry forever in Go (every 1 s); Python gives up after 3 tries and exits. In Go a permanently bad API key produces an infinitely retrying pod that still consumes a worker slot.

3.3 LLM

Python

CustomOpenAILLMService(model="grok-4-1-fast-non-reasoning", api_key=GROK_4_1_FNR_EASTUS_API_KEY, base_url="https://eastusresourcegroup.openai.azure.com/openai/v1", deployment="cureai-grok4fnr-eastus", switch_llm_config=True, model_group='grok-4.1-fast-sales'). Carries user_id+entity_id for LLM logging, with automatic switching via LLMSwitchingService on errors, temperature=0, prompt metadata persisted to chunks.

Go

LLMProcessor uses hard-coded llmModel = "gpt-4.1" against https://api.openai.com/v1/chat/completions with OPENAI_API_KEY. No temperature control, no Langfuse linkage, no fallback model, no per-call entity tracing.
Hard blocker for drop-in Switching the production sales-call from Grok-4.1-fast (an Azure deployment that’s actively price- and quality-tuned for sales) to OpenAI gpt-4.1 will change every conversion-rate, latency, and cost number. LLMSwitchingService also handles model failover; without it any Grok outage today would have nowhere to switch to. This must be parameterised before cutover.

3.4 TTS

Python

CartesiaTTSService(api_key=cartesia_api_key, voice_id="95d51f79-c397-46f9-b49a-23763d3eaa2d", model="sonic-3", params=InputParams(language=Language.HI), text_filters=[PhoneticTextFilter()]). cartesia_api_key comes from the conversation row, allowing per-user keys.

Go

Same voice ID and model literal-embedded in tts_processor.go. No language param, no phonetic dictionary preprocessing, single CARTESIA_API_KEY env var for every call.

Hindi pronunciation will be noticeably worse without PhoneticTextFilter — the phonetic dictionary maps brand names, drug names, and Devanagari spellings into Latin-letter spellings Cartesia pronounces correctly. The language=hi hint also nudges Cartesia’s prosody.

3.5 Conversation context & initial messages

Python

  1. If previous_messages: rebuild as {role, content} filtering is_debug_log and additional_data.
  2. Prepend system prompt with patient_info+current_datetime substitution.
  3. If resume_message: append a fresh system message.
  4. Otherwise (fresh call): seed with system prompt + {user: "hello?"}.
Wraps the result in LLMContext with system_message_prompt_key, prompt_metadata, system_message_params — these are persisted with every chunk.

Go

initialMessagesFromChunks(SalesCallSystemPrompt, startup.Data.Chunks, {user:"hello?"}) exactly mirrors steps 1, 2 (with the hard-coded English prompt), and 4. No resume_message step. No system_message_prompt_key on chunks — the Redis-to-DB sync will write null there.

3.6 Talk-time monitor

Python

TalktimeMonitor arms its timer only after the first UserStartedSpeakingFrame. On timeout: pushes TTSSpeakFrame("Your allocated talk time has been exhausted. Thank you for your time."), calls on_talktime_exhausted (PATCH /update_conversation with end_reason, send RTVI talktime_exhausted, perform_cleanup).

Go

TalkTimeMonitoringProcessor arms its timer in Start (before the user joins). On timeout: pushes InterruptFrame + TTSSpeakFrame("Your talk time is exhausted now. Ending the call.") then calls taskCtx.EndTask(EndReasonTalkTimeExhausted). Sends server-message RTVI talktime_exhausted. End reason maps correctly to Disha.
Behavioural diff Because Go starts the timer at construction, the user’s talktime budget begins counting before they actually speak (or even join). For users with 30 seconds left this can be the difference between a successful upsell and an empty pod. Move the timer arm into ContextAggregator’s first user-message branch (or react to a future UserStartedSpeakingFrame).

3.7 User idle

Python

UserIdleProcessor from Pipecat with timeout=5.0, callback handle_idle that pushes TTSSpeakFrame("hello?", append_to_context=True) up to 6 times and then calls task.cancel() with user_idle_timeout_triggered=True → end_reason user_idle on the post-call. Plus a global idle_timeout_secs=120 on the PipelineTask with cancel_on_idle_timeout=True as a hard safety net.

Go

UserIdleProcessor: 7 s timeout, max 7 prompts. On reaching the cap it just stops prompting — it does not call taskCtx.EndTask(EndReasonUserIdle). No equivalent of the 120 s no-frames-at-all safety net.
Bug-shaped gap A user who silently joins and never speaks will hold the pod for the full talktime budget (default 600 s) before the talk-time exhaustion path kicks in — and they will hear “Hello?” seven times. Python kills the call after ~30 s of silence (5 s × 6). Easy fix: when idlePromptCount reaches the cap call p.taskCtx.EndTask(EndReasonUserIdle).

3.8 RTVI & client messages

Python

CustomRTVIProcessor + CustomRtviObserver emit rtvi-ai entries for bot-transcription, user-transcription, bot-started/stopped-speaking, server-message (metrics, llm_call_result, interruption, talktime_exhausted, participant-left). Also handles inbound {type:"ping"} and responds with "pong" via send_server_response.

Go

UIEventSender emits all the outbound types above through Daily app messages and snapshots them into the S3 debug log. Inbound side handles {type:"end_call"} only — no ping/pong.

If your client relies on pong-based keepalive (look at daily-client.html), it will time out against talk-go. The Disha-shipped browser client does send pings periodically.

4. Pipeline parity (frames, processors, ordering)

Both run the same logical chain:

transport.input() → user_idle → rtvi → stt → user_aggregator → llm → talktime → tts → transport.output() → assistant_aggregator

The Go ordering folds the user and assistant aggregators into one ContextAggregator that lives between STT and TalkTime — functionally equivalent because the Go aggregator commits on TTSDoneFrame arriving upstream from PlaybackSink. Differences worth flagging:

  • Min-words barge-in — both use min_words=3. ✓
  • SpeechTimeoutUserTurnStopStrategy (Python user_speech_timeout=0.05) — talk-go relies on Soniox's <end> token directly; functionally similar but the 50 ms guard does not exist.
  • Back-channel discard — talk-go implements both branches (in-progress and at-bot-turn-end), matching Pipecat reset_aggregation. ✓
  • InterruptFrame purges procCh keeping !IsInterruptible() — mirror of Pipecat UninterruptibleFrame mixin. ✓
  • WordTimestampFrame interleaving — both interleave words with audio frames at PlaybackSink; talk-go uses Cartesia add_timestamps. ✓
  • EndFrame-driven cleanup — both route shutdown through the source so STT/AudioSource/UserIdle/ContextAggregator see it in pipeline order. ✓
  • Custom Daily transport join retry (3×) — Python implements CustomDailyTransportClient.join with 3 retries. talk-go's JoinDailyRoom dials the bridge once with a 20 s join timeout. Add retry if Daily intermittently rejects joins.
  • Background office sound mixer — Python plays background-office-sound.mp3 at volume 0.5 via Pipecat's CustomSoundMixer. talk-go has the mp3 file checked in but, on current reading, doesn't wire it into PlaybackSinkProcessor. Verify before shipping; users notice a quiet bot.

6.4 Why safe-to-evict matters

GKE Autopilot uses the cluster-autoscaler.kubernetes.io/safe-to-evict annotation to decide whether a pod can be evicted during scale-down. Python flips it to false at two moments: when the pod is reserved, and when the pod becomes active for a call. talk-go does neither — if the autoscaler hits the pod during a call, the call dies abruptly. Even adding the annotation only at reservation time would catch most cases (the reservation comes from Disha's reserve_worker_pod before create_worker_room, so the Go pod is reserved for several seconds before the user joins).

7. Signal handling & graceful shutdown

Both implementations follow the same protocol:

  1. First SIGTERM — ignore duplicates, write Redis key pod_sigterm:{pod} with 2 h TTL, enqueue bots.signal_handler.on_graceful_shutdown_initiated on fifo-p0-fast-l1.
  2. Idle pod → exit immediately so Kubernetes rollouts don’t stall.
  3. Active/reserved pod → stay alive; Disha’s state-manager job decides whether to cleanup.
  4. SIGINT → immediate exit for local dev parity.

talk-go’s registerCleanupHandlers matches this. Differences:

  • Python has an atexit handler that emits a Sentry "Abrupt machine kill detected" message when shutdown wasn’t graceful. talk-go has no equivalent.
  • Python uses os.write(2, ...) for signal-safe logging (avoids reentrancy into buffered I/O); Go’s log package is already safe enough that this is not needed.

11. Cutover plan if you decide to ship

  1. Stage in dual-write mode first. Keep Python sales-call serving 100% of traffic. Point Disha's create_room for a tiny user cohort (1%) at the talk-go staging deployment. Capture chunk persistence and post-call rows side-by-side and compare conversion rate and v2v latency for a week.

  4. Run a 1-hour bake on staging with synthetic users (no real Cartesia/Grok billing impact if you mock) to confirm zero goroutine leaks — the captureGoroutineStacks dump should never fire on a clean run.
  5. Promote behind a per-user feature flag that flips CreateRoom in Disha to call talk-go for selected users. Roll out 1% → 10% → 50% → 100% over a week with conversion-rate guardrails.
  6. Decommission Python sales-call only after talk-go has served 100% sales traffic for two weeks with no regressions on (a) chunk persistence completeness, (b) call success rate, (c) p99 v2v latency, (d) Sentry error count.

Replacing the Python sales-call is not a downgrade once M1–M9 are in.
