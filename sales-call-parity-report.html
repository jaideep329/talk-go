Executive Summary

talk-go is a working real-time voice prototype. It already handles browser audio over LiveKit, Opus decode/encode, Soniox STT, streamed OpenAI responses, Cartesia TTS, sentence-level TTS aggregation, paced playback, barge-in interruption, word-timestamp-based assistant commit, idle prompts, a basic live UI, multi-session in-memory routing, and latency metrics.

It is not yet a sales-call product runner. The Python sales-call flow in Disha-Backend is tied into user/conversation state, prompt documents, pricing experiment variants, previous conversation resume, Redis/S3/DB persistence, LLM logging, RTVI client events, scheduled call orchestration, worker pod lifecycle, Daily room lifecycle, talktime limits, post-call operations, and metrics/debug-log upload.

Bottom line: the low-level voice loop is the part closest to parity. The biggest missing pieces are the surrounding application contract: how a call is created, how a conversation_id and user_id enter the session, how prompt/context is loaded, how messages and metrics are saved, and how the backend is notified when the call starts, ends, fails, or exhausts talktime.

Current talk-go Implementation

AudioSourceProcessor → STTProcessor → UserIdleProcessor → ContextAggregator → LLMProcessor → TTSProcessor → PlaybackSinkProcessor

The pipeline is custom-built in Go using data/system channels and directional frame routing. System frames such as InterruptFrame bypass normal buffered data on the downstream path.

  • MissingNo DB, Redis, S3, SQS, Sentry, GKE worker, app notification, recording, or post-call integration.

Documentation drift found: the project instructions describe net/http/pprof in main.go and a 15-second, max-2 idle prompt policy. The current code does not import pprof, and user_idle_processor.go uses 7 seconds with max 7 prompts.

Disha-Backend Sales Call Flow

The Python sales-call implementation is a Pipecat/Daily voice worker plus a backend orchestration layer. The voice worker itself lives in bots/sales_call/sales_call.py, but its behavior depends on reusable modules under bots/onboarding_call, bots/followup_call, bots/services, bots/managers, calling, documents, and services.

Daily input → UserIdle → RTVI → CustomSonioxSTT → LLM user aggregator → CustomOpenAILLM → TalktimeMonitor → Cartesia TTS → Daily output → LLM assistant aggregator

Conversation Bootstrap

  • Fetches CallConversation, user profile, previous chunks, resume data, unprocessed chat context, and Cartesia API key through fetch_conversation().
  • Selects one of three sales prompts based on campaign_pricing_experiment_flag.
  • Renders prompt documents via DocumentManager with patient_info and IST datetime variables.
  • Seeds context with previous messages, a resume system message if needed, or hello? for fresh calls.

Runtime Call Behavior

  • Uses Soniox with Hindi language hints.
  • Uses Cartesia sonic-3, Hindi voice, and PhoneticTextFilter loaded from S3.
  • Uses Pipecat turn strategies: min 3 words to start, 0.05 s speech timeout to stop.
  • Allows interruptions through Pipecat PipelineParams(allow_interruptions=True).
  • Uses user idle prompts every 5 seconds, up to 6 retries, then cancels and marks USER_IDLE.
  • Enforces remaining sales-call talktime and emits a final exhaustion message.

Backend and Ops

  • Updates bot_joined_at, user_joined_at, user_first_speech_at, and bot_first_speech_at.
  • Persists user/assistant chunks to Redis, uploads prompt params/state to S3 where relevant, and queues DB sync.
  • Logs LLM calls with model, prompt metadata, token usage, latency, status, errors, and interrupted-call state.
  • Uploads RTVI/debug logs to S3 and runs post-call operations.
  • Queues Daily metrics collection and worker cleanup.
  • Scheduled calls, callbacks, missed/rejected outcomes, app notifications, and GKE worker reservations are handled outside the worker.

What Is Required For Parity

1. Make Go sessions conversation-aware

Add a session creation API that accepts or receives from Disha Backend: user_id, conversation_id, client_session_id, bot_type=sales_call, room name, tokens, and feature/config flags. A SessionContext should carry this identity along with logger, room, UI events, cancellation, backend client, and session config.

Expected work: replace random local sessions as the production entry path, keep the current debug /connect as a local-only path if useful.

2. Recreate fetch_conversation() behavior

The Go app needs to load conversation and user context before the first LLM turn. The simplest path is not direct DB access from Go; it is a Disha Backend API that returns the same normalized payload the Python worker currently assembles: patient info, previous chunks, resume message, prompt variables, campaign pricing flag, Cartesia API key, remaining talktime, and unprocessed chat context.

Expected work: define a typed Go ConversationBootstrap model and a backend endpoint or internal service call that produces it.

3. Replace the hard-coded prompt with document-backed sales prompts

Prompt parity requires selecting the prompt by campaign_pricing_experiment_flag and rendering it from Disha's document system. At minimum, Go needs either a prompt-render API or a pre-rendered prompt in the bootstrap payload, plus prompt name/version metadata for logging.

Expected work: avoid copying prompt text into Go. Keep Disha Backend as the source of truth for prompt documents.

4. Add production persistence

Create a Go equivalent of ConversationPersistenceProcessor. It should save user and assistant turns with timestamps, role, conversation ID, user ID, bot type, metrics, prompt key, and additional/debug data. The most practical implementation is to call Disha Backend APIs or a queue-backed ingestion endpoint rather than writing Redis/S3/DB logic directly in Go.

Expected work: persist every committed user turn and every assistant turn, including interrupted partial assistant turns if they are user-audible.

5. Implement lifecycle event reporting

The Go worker needs to report the same lifecycle timestamps that the Python worker reports: bot joined, user joined, first audio frame received, first user speech, first bot speech, user left, ended at, total user duration, and any end reason. For LiveKit, these events map to LiveKit room/participant callbacks and local pipeline events.

Expected work: add idempotent cleanup and update calls to Disha Backend, including failure paths.

6. Add talktime admission, monitoring, and deduction

Sales calls are gated by remaining_sales_call_talktime_seconds. Go needs a pre-call check, a runtime monitor starting on first user speech, a final exhaustion utterance, call cancellation/cleanup, and post-call deduction through the existing backend operations.

Expected work: implement a Go TalktimeMonitorProcessor or session-level monitor equivalent to Python's TalktimeMonitor.

7. Align idle and turn-ending behavior

Current Go interruption logic is close, but idle behavior and normal turn stopping differ. Make idle timeout, retry count, prompt text, and "end on exhausted idle retries" configurable. Decide whether Soniox endpointing alone is acceptable or whether Go needs a separate speech-timeout strategy equivalent to Pipecat's 0.05 s stop rule.

8. Add phonetic filtering and TTS cleanup

The Python TTS path applies PhoneticTextFilter, removes accidental function-update text, and drops non-speakable output. Go should apply the same text filter before sending text to Cartesia, ideally by loading a phonetics dictionary from the same source or receiving a preloaded dictionary/config from backend.

9. Add LLM configuration, failover, and logging

Parity needs Disha's model-selection behavior: model group, config key, deployment, provider credentials, slow-call polling, error blacklist/reselection, token usage, cached tokens, interrupted-call logs, prompt metadata, and status code capture. If that is too much for the first version, keep model switching in Disha Backend and expose an OpenAI-compatible proxy endpoint to Go.

10. Define LiveKit replacements for Daily-specific features

Disha-Backend currently has livekit_room_name, livekit_recording_url, and transport_type fields, but the active sales-call path is Daily. If the product is moving to LiveKit, build the missing LiveKit equivalents for room creation, client token generation, recording, metrics, client join failures, client hangup/end reason, and worker assignment.

Recommended Migration Shape

Recommended approach: keep Disha Backend as the orchestrator and make talk-go a production voice worker behind it. Do not port scheduling, DB models, SQS jobs, document storage, or post-call business logic into Go unless there is a separate reason to split those systems.
  1. Backend owns call creation. Disha Backend creates or reuses the CallConversation, checks talktime, assigns a worker, creates the LiveKit room/token, and forwards a worker-start request to the Go service.
  2. Go owns realtime media. The Go worker joins LiveKit, runs STT/LLM/TTS/playback, detects turns and interruptions, and emits lifecycle/message/metric events.
  3. Backend owns durable side effects. Go sends structured events to backend APIs. Backend persists chunks, logs, prompt metadata, post-call operations, talktime deductions, scheduled-call outcomes, and analytics.
  4. Frontend gets a transport-specific connection payload. For LiveKit, the client needs LiveKit URL/token/room plus the same client-session semantics currently used by Daily.

Highest-Risk Gaps

Risk Why It Matters Mitigation
Transport migration from Daily to LiveKit Daily is not just audio transport in the current flow; it also carries RTVI messages, participant events, metrics, and recording assumptions. Write an explicit LiveKit feature map before replacing Daily. Include recording, client failure reporting, app messages, participant lifecycle, and metrics.
Durability during interruptions and disconnects The current Go app can lose in-memory messages if the process dies or cleanup races. Sales calls need durable user/assistant chunks and post-call metadata. Persist turn commits immediately and make cleanup/post-call operations idempotent.
Prompt/context mismatch Even perfect audio behavior will not be sales-call parity if the prompt, previous context, resume instruction, and pricing variant differ. Have Disha Backend render the exact prompt and pass prompt metadata to Go.
LLM provider behavior The Python path uses model switching, specific model group semantics, and detailed LLM logging. A hard-coded OpenAI model changes both behavior and observability. Initially call a backend LLM proxy or port the config-selection subset required for live sales calls.
Talktime and call outcomes Sales calls are business-gated. Missing talktime enforcement can overrun paid/free limits and produce inconsistent user state. Add both pre-call admission and in-call runtime enforcement before production traffic.

Open Decisions

  • LiveKit is assumed but not yet productized. The Go app is LiveKit-first, while the current Disha sales-call product is Daily-first. Confirm that replacing Daily is the intended migration, not just a Go rewrite.
  • Backend API surface needs design. The cleanest path is to add backend endpoints for bootstrap, turn persistence, lifecycle updates, and post-call operations instead of giving Go direct DB/Redis/S3 responsibility.
  • Prompt rendering ownership. Backend should probably render prompt documents and pass prompt metadata to Go so prompt rollout/versioning stays centralized.
  • LLM routing ownership. Either port the Disha model-switching service to Go or expose an OpenAI-compatible internal endpoint that already applies Disha routing/logging policy.
  • RTVI replacement. If the Disha clients depend on RTVI semantics, LiveKit data messages or a side websocket must carry equivalent status/debug/server messages.
